## Code of Conduct

This project adheres to the [Contributor Covenant](CODE_OF_CONDUCT.md). By participating you are expected to uphold it. Report unacceptable behavior to the maintainers.

## Ways to contribute

- **Fix a bug** — pick something from [open bugs](https://github.com/nixrajput/pg-sync/labels/bug).
- **Improve docs** — typos, clearer explanations, missing examples.
- **Add a feature** — open a feature request issue first so we can agree on scope before you spend time coding.
- **Test on new platforms** — especially Linux distros, WSL, Git Bash.
- **Write tests** — the smoke test in `tests/01_smoke.sh` is intentionally small; we welcome more coverage.

## Getting started

You need:

- **Bash 3.2+** (every macOS and Linux has this).
- **Git**.
- **PostgreSQL client tools** (`pg_dump`, `pg_restore`, `psql`) — needed to run the script end-to-end. The smoke tests don't require them.
- **shellcheck** for lint: `brew install shellcheck` / `apt install shellcheck`.
- **A local PostgreSQL** if you want to test against a real DB.

Clone and run the dev loop:

```bash
git clone https://github.com/nixrajput/pg-sync.git
cd pg-sync

make lint    # bash -n + shellcheck
make test    # smoke tests
make build   # produce dist/ artifacts
make install # install your working copy to ~/.local/bin
```

After `make install`, `pg-sync --help` runs your local version.

## Development workflow

A typical change:

1. **Open an issue first** if your change is non-trivial (>50 LOC or any user-visible behavior change). For typo fixes or tiny refactors, skip straight to a PR.

2. **Fork and branch** from `main`:

   ```bash
   git checkout -b feat/your-feature
   ```

   Branch naming is flexible; we don't enforce a strict convention. Common prefixes: `feat/`, `fix/`, `docs/`, `refactor/`, `test/`.

3. **Make your change**, keeping the [code style](#code-style) in mind.

4. **Add or update tests** under `tests/` if your change affects behavior that can be tested without a live database.

5. **Update `CHANGELOG.md`** under the `[Unreleased]` heading. Use one of the Keep-a-Changelog categories: `Added`, `Changed`, `Deprecated`, `Removed`, `Fixed`, `Security`.

6. **Run the dev loop locally before pushing**:

   ```bash
   make lint test build
   ```

   All three must pass. CI runs the same commands.

7. **Open the PR** using the template. Link the issue with `Closes #NN` if applicable.

8. **Address review** — push additional commits to the same branch; the reviewer will squash on merge.

## Code style

`pg-sync` is bash. We follow these conventions:

- **Indent with 4 spaces.** No tabs.
- **Use `[[ ]]` for tests**, not `[ ]`.
- **Quote variable expansions**: `"$var"` not `$var`, unless intentionally word-splitting.
- **Functions are named with `lower_snake_case`**.
- **Use `local` inside functions** for everything except intentional globals.
- **No `set -e` inside diagnostic background loops** — they need to tolerate failing subcommands (e.g. `pgrep` returning 1 for "no matches"). The script-level `set -euo pipefail` is non-negotiable.
- **All logging goes to stderr** (`log_info`, `log_warn`, `log_ok`, `log_error`, `log_step`, `banner`). Functions that return values use `printf '%s' "$value"` to stdout. This keeps `result=$(my_helper)` clean.
- **Prefer named functions** over long inline `case` branches.
- **Comment the _why_, not the _what_**. Surprises and constraints deserve comments; obvious code does not.

### shellcheck must pass at `-S warning`

Before pushing:

```bash
shellcheck -S warning src/pg-sync
```

If a warning is a genuine false positive, you can disable it for a single line with `# shellcheck disable=SCXXXX` and a brief justification, e.g.:

```bash
# shellcheck disable=SC2086  # intentional word-splitting on arg list
pg_dump $extra_args
```

Don't blanket-disable shellcheck warnings.

## Commit messages

We don't enforce Conventional Commits, but follow these rules:

- **Subject line ≤72 characters**, imperative mood: "Add foo" not "Added foo" or "Adds foo".
- **Blank line before the body**, body explains _why_ (not what — the diff shows what).
- **Reference issues** in the body or trailer: `Closes #42` or `Refs #42`.

Example:

```
Add per-table restore mode

When users only need to refresh one table (e.g. `users`), the previous
flow forced a full restore. This adds a `per-table` restore mode that
calls pg_restore --table=NAME, supporting both schema+data and data-only
variants.

Closes #17
```

## Pull request checklist

Use the [PR template](.github/PULL_REQUEST_TEMPLATE.md). It boils down to:

- [ ] `make lint` passes
- [ ] `make test` passes
- [ ] `make build` produces a working tarball
- [ ] `CHANGELOG.md` updated under `[Unreleased]`
- [ ] New or changed behavior documented in `README.md` if user-visible
- [ ] PR description explains the **why**, not just the **what**
- [ ] Linked an issue (or explained why not)

## Releases

Cutting releases is a **maintainer-only** task. The full runbook lives in [`RELEASING.md`](RELEASING.md).

If you're a contributor, you don't need to do anything release-related — just add your changes under `## [Unreleased]` in `CHANGELOG.md` and the maintainer will roll them into the next release.

---

## Reporting bugs

Use the [bug report template](https://github.com/nixrajput/pg-sync/issues/new?template=bug_report.yml). Include:

- `pg-sync --version` output
- OS + version (e.g. macOS 14.4, Ubuntu 22.04, WSL2)
- Exact command you ran
- What you expected
- What actually happened (paste relevant log lines)
- Any redacted heartbeat output if relevant

For security issues, **do NOT open a public issue**. See [`SECURITY.md`](SECURITY.md) for the disclosure process.

---

## Asking questions

For usage questions, prefer [GitHub Discussions](https://github.com/nixrajput/pg-sync/discussions) over issues. Issues are for bugs and concrete feature requests.

Thanks for contributing! 🎉
